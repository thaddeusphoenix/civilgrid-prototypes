Account-level access — Projects & Access
Purpose

A new top-level Account entity sits above a customer's Orgs and is reached from a new Account Settings item in the profile menu. Its main job is to house access to all projects across every org. This prototype puts three structures for that surface behind an A / B / C toggle — the open question is which one to build.

What to notice
  • Open the NS avatar (top right) — the new Account Settings item is the entry point into this console.
  • Everything is one fictional enterprise (Northwind Energy, 4 orgs). The A/B/C toggle changes only how Projects & Access is structured — same underlying data.
  • A · Direct grants — one flat table of every project; grant access per project (gear or click the access cell), or select rows and bulk-add. Projects can be shared across orgs (the duplicate-work / duplicate-billing case).
  • B · Group cascade — Account → Org Group → Org → Project tree; Grant at any node and it cascades down. Projects show where their access came from (via Account, via Land Services…).
  • C · Attribute rules — access is rules over project attributes (Region / Division / Customer / Project Type — the same fields as member-access-scope-b006), matched automatically. Add a rule and watch the matching-project count.

If we build this — Linear issues
Foundation / data model
  • Introduce Account as the top-level entity above Orgs; one Account per enterprise buyer
  • Link existing Orgs to a parent Account; migrate multi-org customers with many separate orgs under one Account
  • Account-admin role: manage orgs, people, and access across the Account without CivilGrid staff
  • (A) Allow a project to be homed in more than one Org; resolve billing attribution for shared projects
  • (B) Add an Org Group tier; model Account → Org Group → Org; define grant inheritance / cascade
  • (C) Tag every project with structured attributes (Region, Division, Customer, Project Type); backfill existing projects
Account console UI
  • Add "Account Settings" to the profile menu; build the Account console shell (Overview / Projects & Access / People / Orgs)
  • (A) All-projects table across orgs with per-project people grants + bulk apply
  • (B) Collapsible Account→Group→Org→Project tree; grant at any node; show inherited access on children
  • (C) Access-rule builder over project attributes with a live matching-project preview and counts
  • Account Overview rollup (orgs, project counts, members) for the economic buyer
Enforcement / in-app (the real change)
  • Filter project visibility across orgs by the chosen access model, server-side
  • Cross-org project query + duplicate-boundary detection before a project is created
  • Access lifecycle for external grants — expiry, revoke, audit trail (Access Grant object)
Decisions & spikes
  • Which structure — direct grants (A) vs. group cascade (B) vs. attribute rules (C)? (this prototype's question)
  • Account alone, or Account + Org Group as the tier? (product + engineering; architecture decision)
  • Does Account change billing / the duplicate-charge problem for shared projects? (open — product + eng)
  • Is "Org Group" the right customer vocabulary — Division / Region / Portfolio? (open — customer calls)
  • Does access travel with the person, or with a role that carries a default scope? (open)
