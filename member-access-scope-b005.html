Member Access Scope — Members
Purpose

Where should access control live? This puts it on the member: a person gets a Role (what they can do) plus an Access Scope (which projects they can see) — the structured generalization of today's "all projects" vs. hand-picked Guest "subset of projects."

What to notice
  • The Members page is reproduced from the real source (Organization Members / Guest Members / Pending / CivilGrid Support).
  • New Access Scope column — most members are "All projects," some are scoped by the four structured fields.
  • Access tab: the admin page that defines the four fields (Region / Division / Customer / Project Type) and their allowed values — what the invite flow and scope editors pick from. Add/remove values inline.
  • Invite flow (focus): "Invite Team" opens the real production dialog (multi-email add) extended with Role + Access Scope, so invitees arrive pre-scoped. They land in Pending invites carrying that scope.
  • Hover a member → the gear opens Edit User: Role dropdown (unchanged) + a new Access Scope section with a live "this person can see…" summary.
  • Guest Members' hand-picked project list becomes structured scope — the same model, just tighter.

If we build this — Linear issues
Foundation / data model
  • Add structured scope fields (Region, Division, Customer, Project Type) with admin-defined values
  • Anchor Region values to geospatial service-area polygons
  • Tag every project with its four field values; backfill existing projects
  • Add per-member access scope; migrate Guest "subset of projects" to structured scope
Members UI
  • Add Access Scope column to the member tables
  • Add Access Scope section to the Edit User dialog (per-field multi-select + live summary)
  • Bulk-select members + apply one scope to all selected
  • Set scope at invite time in the Invite Team flow
Enforcement / in-app (the real change)
  • Filter project visibility by each member's scope server-side
  • Cross-region query support ("all gas transmission in the US")
  • Migration: collapse a customer's multiple orgs into one + map old orgs to member scope
Decisions & spikes
