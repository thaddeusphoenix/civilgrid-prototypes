Feature Provisioning — Org Subscription
Purpose

Pressure-test which features should be toggleable per project tier — and how a per-org override unblocks a real customer case (downloads + potholing without CivilSearch). From a June grooming call.

Linear project

Admin Panel - Customize Organization Project Tier Subscription w/ Features ↗

What to notice
  • Features × tiers (Draft / Project Light / Standard / Plus) for this org. Tier defaults are assumed — they're not a separate view; this screen only edits what this org overrides.
  • Acme runs on tier defaults with a few overrides baked in (downloads + potholing on, CivilSearch off in Project Light). A lead-in states the override model; an override count + Save sit above the grid.
  • Overridden cells flag amber and the count updates live; everything else keeps following the default.
  • Per-org tier description — under each tier, "+ Add description" sets a blurb for this org (meant to surface to the org's users in-app). The tier name stays fixed — it's a canonical product object, not renamed per org.
Not real yet

Tier defaults shown are a first guess — that's the open question for product & engineering. No save/persistence. The Feedback button submits to a live Google Form (stamped with prototype + build + screen).
