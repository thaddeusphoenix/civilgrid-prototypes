Contract Credits — Admin
Purpose

Give admin a way to set an org's pre-purchased Project Pool (project count) and contract term by linking a Salesforce Opportunity — the pool and term pull from Salesforce instead of freeform entry, so what admin sets always matches what was sold. The input side of the customer-facing Usage view (contract-prebuy-usage).

Linear project

Billing & Fee Transparency ↗

What to notice
  • Salesforce-linked (b004). Instead of freeform entry, admin picks or links a Salesforce Opportunity: the Look up action pulls the Project Pool (project count) and contract term straight from Salesforce. The pool is read-only — it always comes from the Opportunity, so admin can't silently diverge from what was sold.
  • Dates are an override. Contract Start/End default to the Salesforce values but stay editable — for the utility edge case where projects are loaded/initiated before the contract start date. Overriding a date shows an "overridden" tag + a Reset to Salesforce dates link; the pool is never touched.
  • Multiple contracts per org. The Contract selector at the top switches between an org's contracts (each tied to one Opportunity), or links a new one from Salesforce. Pairs with the customer-facing per-contract Usage view.
  • The pool is denominated in Project Units (project count) — grounded in CivilGrid's real Order Form pricing (Project Unit = 0.5 mi or 20 acres per project; contracts are sold as e.g. "100 Project Units"). A status strip (projects created, pool, remaining, term) sits above so admin has context.
  • Save confirms the org's Usage view now reflects these values — this is exactly what contract-prebuy-usage reads from. Overage Notifications — toggle + recipient list so account owner(s) get emailed when this org exceeds its pool.
  • Lives on the same Update Organization Subscription admin page as Pricing and Feature Provisioning (same tab bar) — those two are stubbed here to keep the focus on Contracts.
Not real yet

Acme, Inc. Organization, its numbers (42 created / 100 pool), and the Opportunity IDs are fictional but scaled to the real Jan 2026 Order Form example. The Salesforce "lookup" doesn't call Salesforce — it loads canned per-contract data. No save/persistence; the email isn't wired (real send would go through Knock). The Feedback button submits to a live Google Form (stamped with prototype + build + screen).

If we build this — Linear issues
Foundation / data model
  • Model a Contract tied to a Salesforce Opportunity (store Opp ID → Project Pool + term); support multiple contracts per org
  • Define how projects-created is computed/rolled up from project activity (1 project = 1 Project Unit, up to 0.5 mi / 20 acres; oversized projects consume more than one)
  • Store both the Salesforce-sourced dates and any admin override (so provenance + reset are possible)
Salesforce sync
  • Look up an Opportunity by ID and pull Project Pool + contract term
  • Map which Salesforce fields feed Project Pool and contract start/end (spike)
  • Keep the pool read-only (Salesforce-sourced) while allowing date overrides — never overwrite the pool on override
  • Re-sync + last-synced provenance; handle Opportunity-not-found / not-yet-linked
Admin: Contracts (per-org)
  • Add Contracts tab to Update Organization Subscription
  • Add Contract selector + "Link new contract from Salesforce"
  • Opportunity lookup UI; read-only pool display + editable date-override fields with "overridden" state + reset
  • Current-status readout (projects created / pool / remaining / term)
  • Overage Notifications section: enable/disable toggle + recipient list (add/remove emails)
  • Persist + save Contracts tab (dirty state, confirmation) — including notification settings
Enforcement / in-app
  • Feed the saved pool + term into the org's customer-facing Usage view
  • Surface remaining pool on project create
  • Detect when an org's project count crosses its Project Pool (the trigger for the email below)
  • Wire a Knock workflow + email template for the pool-exceeded notification; send to the saved recipient list
Decisions & spikes
  • Reconcile with the delivery app — CS flagged that a lot of this usage tracking already exists in the delivery app and should move to this surface, not be duplicated; scope the migration before building (CS, Jul 21)
  • Decided b004: contract setup is Salesforce-driven — pool + term come from the Opportunity (no freeform); dates are an override for the utility early-load case (CS, Jul 21)
  • Decided b002: pool denominated in Project Units (project count), grounded in the real Order Form pricing model — not raw acreage/mileage, not committed $ (PM)
  • Decided b003: overage gets an email notification to account owner(s) — notification only, not billing; the pool doesn't hard-stop project creation (PM)
  • Overage billing still not modeled — real contracts already price per-project overage, but charging for it is separate from notifying (see Jan 2026 Order Form)
  • Are "account owner(s)" the customer's own billing contact, or CivilGrid's internal AE/CSM? Recipient list is free-text either way — confirm who gets this email (open)
