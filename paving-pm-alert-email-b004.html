Paving Data-Refresh Alert — PM Email
Purpose

Does this email's shape and content give a utility project manager enough to triage a paving/moratoria change within their project boundary? Built to react to, not to ship. (Customer-specific data is fictional.)

Linear project

Conflicts Data Added Email Alert ↗

What to notice
  • One email per project, per PM — filterable by project, no digest
  • Trigger is any new/changed data inside the project boundary — not only conflicts
  • Deliberately modeled on the existing change-notification email so it reads as familiar
  • Bullets carry just enough to triage; "Review Changes" deep-links to the project

Foundation / change detection
  • Diff each paving/moratoria refresh against prior loaded state per jurisdiction ID
  • Persist prior moratorium state to support feature-level deltas (not just "data refreshed")
  • Spatial join: match refreshed jurisdiction data to active project boundaries (+ buffer)
Notification pipeline
  • Add per-project paving-change email template
  • Emit one email per project, per assigned PM, on detected change
  • Add per-org feature flag for paving-alert email sending (e.g. a land/right-of-way org = off)
Recipients & preferences
  • Resolve project → assigned PM(s) for addressing
  • Per-user notification preferences (opt-in / opt-out link target)
Enforcement / in-app
  • "Review Changes" deep link → project change view
Decisions & spikes
  • Can the rebuilt pipeline surface which field changed, or only that data refreshed? (feasibility spike)
  • Moratorium moved but still no conflict — alert or suppress?
  • Strict project boundary vs. ~½-mile buffer
  • Confirm one-email-per-project over digest on initial city load
